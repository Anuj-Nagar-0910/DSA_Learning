1. Current Salary Of Department Managers
INNER JOIN Aliases Active Salary

Task

Retrieve employee number, first name, last name, department name, current title, and current salary of department managers where salary is active.

Final Query

SELECT
    e.emp_no,
    e.first_name,
    e.last_name,
    d.dept_name,
    t.title AS current_title,
    s.salary AS current_salary
FROM employees AS e
INNER JOIN dept_manager AS dm
    ON e.emp_no = dm.emp_no
INNER JOIN departments AS d
    ON dm.dept_no = d.dept_no
INNER JOIN titles AS t
    ON e.emp_no = t.emp_no
INNER JOIN salaries AS s
    ON e.emp_no = s.emp_no
WHERE s.to_date = '9999-01-01';

Why It Works

  • `salaries` stores salary history.
  • `s.to_date = '9999-01-01'` keeps active salary rows.
  • Multiple `INNER JOIN`s bring together details from separate normalized tables.

Shortcoming

If the prompt asks for current manager and current title, the query may also need filters on `dept_manager.to_date` and `titles.to_date`.

2. Employees Not Assigned To Sales
INNER JOIN DISTINCT WHERE

Final Query

SELECT DISTINCT
    e.emp_no,
    e.first_name,
    e.last_name
FROM employees AS e
INNER JOIN dept_emp AS de
    ON e.emp_no = de.emp_no
INNER JOIN departments AS d
    ON de.dept_no = d.dept_no
WHERE d.dept_name <> 'Sales';

Why It Works

  • The expected result treated the problem as row-level non-Sales assignment rows.
  • `dept_emp` can contain multiple assignment rows per employee.
  • `DISTINCT` keeps each employee only once after joining assignment rows.

What Failed

`NOT EXISTS` was too strict because it excluded anyone who ever had a Sales assignment. Adding `to_date = '9999-01-01'` assumed current assignment, which the prompt did not say.

3. Development Department Tenure
INNER JOIN DATEDIFF No DISTINCT

Final Query

SELECT
    e.first_name,
    e.last_name
FROM employees AS e
INNER JOIN dept_emp AS de
    ON e.emp_no = de.emp_no
INNER JOIN departments AS d
    ON de.dept_no = d.dept_no
WHERE d.dept_name = 'Development'
  AND DATEDIFF(de.to_date, de.from_date) > 365 * 3;

Why It Works

  • `dept_emp.from_date` and `dept_emp.to_date` store department tenure.
  • `DATEDIFF` measures the actual duration in days.
  • No `DISTINCT` is used because different employees can have the same name.

What Failed

`TIMESTAMPDIFF(YEAR, ...)` did not match the expected day-based duration. `DISTINCT first_name, last_name` collapsed valid employees with the same name.

4. Department Salary Expenditure, 1990 To 2000
INNER JOIN SUM GROUP BY Date Range

Final Query

SELECT
    d.dept_name,
    SUM(s.salary) AS total_salary
FROM departments AS d
INNER JOIN dept_emp AS de
    ON d.dept_no = de.dept_no
INNER JOIN salaries AS s
    ON de.emp_no = s.emp_no
WHERE s.from_date >= '1990-01-01'
  AND s.to_date <= '2000-12-31'
GROUP BY d.dept_name;

Why It Works

  • `salaries` does not store department number, so `dept_emp` is needed.
  • `SUM(s.salary)` calculates expenditure.
  • Both date boundaries are checked so the salary row is fully inside the requested period.

What Failed

Filtering only `s.from_date BETWEEN '1990-01-01' AND '2000-12-31'` included salaries that continued after the period, which made totals too high.

5. High-Spending Users At High-Rated Restaurants
CREATE VIEW INNER JOIN SUM GROUP BY HAVING Subquery AVG

Final Query

CREATE VIEW high_rated_restaurants AS
SELECT
    restaurant_id,
    name,
    rating
FROM Restaurants
WHERE rating > 4;

SELECT
    o.user_id,
    SUM(o.total_price) AS total_spending
FROM Orders AS o
INNER JOIN high_rated_restaurants AS hrr
    ON o.restaurant_id = hrr.restaurant_id
GROUP BY o.user_id
HAVING SUM(o.total_price) > (
    SELECT AVG(o2.total_price)
    FROM Orders AS o2
    INNER JOIN high_rated_restaurants AS hrr2
        ON o2.restaurant_id = hrr2.restaurant_id
);

Why It Works

  • The view keeps only restaurants where `rating > 4`.
  • `INNER JOIN` keeps only orders placed at those restaurants.
  • `GROUP BY o.user_id` creates one total per user.
  • `HAVING` filters users after `SUM(o.total_price)` is calculated.
  • The subquery uses `AVG(o2.total_price)`, so the threshold is average order spending, not average user total spending.

What Failed

Selecting `rating` changed the expected output shape and forced grouping by rating, which split user totals. Averaging user-level totals also created the wrong threshold.

Revision Checklist

  • Use `WHERE` before grouping to filter raw rows.
  • Use `HAVING` after grouping to filter aggregate values.
  • Use `DISTINCT` only when selected columns are safe to deduplicate.
  • For history tables, check whether the prompt asks for current rows.
  • For date ranges, check whether rows must start inside, overlap, or be fully contained.
  • Match expected output columns exactly. Extra columns can change grouping and results.
  • In subqueries, confirm whether the average is per row, per order, per user, or per group.
