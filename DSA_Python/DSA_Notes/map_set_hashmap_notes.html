Map, Set, HashMap, HashSet — Python Notes

Intuition — real-life analogies

Map / dict

Phonebook

Name (key) → Phone number (value). You look up a name, instantly get the number. No scanning through all entries — jump straight to it.

Key insight: pairs of (key, value). Every key is unique. Values can repeat.

Set

Guest list

A list of unique attendees. You care only about membership — is this person invited? No duplicates. No order matters.

Key insight: uniqueness only. No values, no order, just existence.

When to use which?

QuestionUseWhy
"Does X exist?"SetO(1) membership check
"What is the value for X?"DictKey → value lookup
"How many times does X appear?"DictFrequency counting
"Find duplicates / unique items"SetAuto-deduplication
"Group items by category"DictKey = category, val = list
"Common items between two groups"Setintersection()

The core rule: if you need to associate a value with a key → dict. If you only need to know presence/absence → set.

Definitions

Map (Python: dict)

A collection of key-value pairs where each key is unique. Implemented internally as a hash table.

# Python dict = HashMap phonebook = {"Alice": 9876, "Bob": 1234} # key="Alice", value=9876
Propertydict
Keys unique?Yes — duplicate key overwrites
Values unique?No — duplicates allowed
Ordered?Yes (insertion order, Python 3.7+)
Mutable?Yes — add, update, delete freely
Key type requirementMust be hashable (immutable)

Set (Python: set)

An unordered collection of unique elements. Internally, it's a hash table that only stores keys (no values).

guests = {"Alice", "Bob", "Alice"} print(guests) # {'Alice', 'Bob'} — deduped
Propertyset
Elements unique?Yes — automatically
Ordered?No — no guaranteed order
Mutable?Yes (use frozenset for immutable)
Element typeMust be hashable
HashMap

Conceptual name used in CS/Java. In Python, dict IS the HashMap. Same O(1) operations, same hash table internals.

HashSet

Conceptual name used in CS/Java. In Python, set IS the HashSet. A hash table storing only keys.

Java/C++ say HashMap/HashSet. Python says dict/set. They're the same concept — tell interviewers you know both.

How it works (high-level)

A hash function converts any key into an integer index. That index tells Python exactly which bucket to put/find the value — no searching needed.

# Python internally does something like: index = hash(key) % num_buckets table[index] = value # O(1) — direct placement

Practice problems

Two Sum

Easy

Given nums and target, return indices of two numbers that add to target.

def twoSum(nums, target): seen = {} # {value: index} for i, n in enumerate(nums): complement = target - n if complement in seen: # O(1) lookup return [seen[complement], i] seen[n] = i return [] # Dry run: nums=[2,7,11,15], target=9 # i=0: n=2, comp=7, seen={} → add {2:0} # i=1: n=7, comp=2, seen={2:0} → FOUND! return [0,1]

Time: O(n) — Space: O(n)

Longest substring without repeating characters

Medium
def lengthOfLongestSubstring(s): seen = {} # char → last seen index l, res = 0, 0 for r, ch in enumerate(s): if ch in seen and seen[ch] >= l: l = seen[ch] + 1 # shrink window seen[ch] = r res = max(res, r - l + 1) return res # Dry run: s="abcabcbb" # r=0:'a' l=0 seen={a:0} res=1 # r=1:'b' l=0 seen={a:0,b:1} res=2 # r=2:'c' l=0 res=3 # r=3:'a' → dup! l=1, seen={a:3,...} res=3 # ... final res=3 ✓

Time: O(n) — Space: O(min(n, alphabet))

import heapq from collections import Counter def topKFrequent(nums, k): freq = Counter(nums) # {num: count} # min-heap of size k return heapq.nlargest(k, freq.keys(), key=freq.get) # nums=[1,1,1,2,2,3], k=2 # freq={1:3,2:2,3:1} # nlargest(2,...) → [1,2] ✓ # Time: O(n log k) Space: O(n) # Bucket sort approach → O(n) def topKFrequent_On(nums, k): freq = Counter(nums) buckets = [[] for _ in range(len(nums)+1)] for num, cnt in freq.items(): buckets[cnt].append(num) res = [] for b in reversed(buckets): res.extend(b) if len(res) >= k: return res[:k]

Optimization insights

Prefer defaultdict(int) over d.get(k, 0) — cleaner for frequency counting.

Use Counter for frequency — it has built-in .most_common(k).

Convert a list to set for O(1) membership: lookup = set(my_list) then x in lookup.

dict consumes ~8x more memory than a list of same size. Avoid for memory-critical applications.

In interviews: always mention O(1) average lookup as the key reason for choosing dict/set. Interviewers love hearing this explicitly.
